# Agent Development Lifecycle Guide

> **Audience**: Agent Engineers, Cognitive Systems Architects, Domain Stewards  
> **Last Updated**: 2026-01-11

---

## Overview

This guide walks through the complete lifecycle of developing, validating, and deploying an AI agent in Seer. The lifecycle mirrors how organizations hire and develop human employees—making it intuitive to understand and follow.

By the end of this guide, you will be able to:

- Conceive an agent grounded in a specific Scenario context
- Train and evaluate an agent in a sandbox environment
- Publish a validated Training Specification for reuse
- Employ an agent to production Workbenches

---

## Quick Start: Deploy Your First Agent

Already have a Scenario and sandbox ready? Here's the minimal workflow:

```bash
# 1. Review the Scenario you're implementing
hub get scenario-normative routine-dispute-triage

# 2. Create your specs (TrainingSpec, HubApplicationSpec, ScenarioAutomationSpec)
# Edit files in VS Code, then commit to Git (GitOps requirement)
git add training-spec.yaml hub-application-spec.yaml scenario-automation-spec.yaml
git commit -m "feat: initial agent specs"
git push

# 3. Create sandbox deployment spec
# Edit scenario-deployment-sandbox.yaml, then commit
git add scenario-deployment-sandbox.yaml
git commit -m "feat: sandbox deployment spec"
git push

# 4. Deploy to sandbox (smallest unit of deployment = Scenario)
# Syncs entire scenario from committed Git files
hub sync scenario routine-dispute-triage-sandbox

# 5. Monitor and iterate
hub logs agent dispute-triage-emp-001 --follow

# 6. When ready, validate and publish
hub validate training-spec dispute-triage-agent-v1 --workbench acme-disputes-sandbox
hub publish training-spec dispute-triage-agent-v1 --version 1.0.0

# 7. Deploy to production
# Edit scenario-deployment-prod.yaml, then commit
git add scenario-deployment-prod.yaml
git commit -m "feat: production deployment spec"
git push
hub sync scenario routine-dispute-triage-prod
```

**New to this?** Continue reading for the complete walkthrough with explanations.

## The Human Analogy

Seer's agent lifecycle directly parallels human employment. This analogy is intentional — it makes the concepts intuitive and leverages familiar organizational patterns.

| Human Employment | Seer Agent Lifecycle |
|------------------|----------------------|
| **Skilled Person** (educated, has competencies) | **Raw Agent** (base capabilities, no domain knowledge) |
| **Trained Employee** (trained for role, knows procedures) | **Trained Agent** (knowledge bound, skills developed) |
| **Employed Worker** (assigned to team, authorized to act) | **Employed Agent** (deployed to Workbench, granted authority) |
| **Training Program** (develops skills for a role) | **Training Specification** (configures agent for Scenario) |
| **Job Assignment** (role in specific project/department) | **Employment Specification** (agent in specific Workbench Instance) |
| **Project/Department** (concrete work context) | **Workbench Instance + Environment** (production context) |
| **Training Facility** (safe environment to learn) | **Sandbox Workbench** (safe environment to validate) |

### The Core Insight

> **Deployment = Employment**
> 
> In Seer, deploying an agent means **employing** it — assigning a Trained Agent to a specific Workbench Instance with a concrete environment, where it can assume roles and complete tasks. Just as skilled humans are *employable* but only become *employed* when assigned to a specific job, Trained Agents are *deployable* but only become *Employed Agents* when bound to a specific operational context.

## What You Create vs. What's Generated

> **Critical for newcomers**: Not all CRDs are created by you. Some are generated by operators.

| CRD | Created By | Your Responsibility? |
|-----|------------|---------------------|
| **ScenarioNormativeSpec** | Process Architect | ❌ You receive this |
| **TrainingSpec** | Agent Engineer | ✅ **You create this** |
| **HubApplicationSpec** | Developer | ✅ **You create this** |
| **ScenarioAutomationSpec** | Developer | ✅ **You create this** |
| **ScenarioDeploymentSpec** | Supervisor | ⚠️ You may create (sandbox) or request (production) |
| **HubApplicationDeployment** | Hub Operators | ❌ Auto-generated |
| **EmploymentSpec** | Seer Operator | ❌ Auto-generated |

**Bottom line**: As an Agent Engineer, you primarily create **TrainingSpec**, **HubApplicationSpec**, and **ScenarioAutomationSpec**. Everything else is either input you receive or output that's generated.

## Prerequisites

Before you begin, ensure you have:

- [ ] An approved Scenario Definition (from Process Architect)
- [ ] A Cognitive Design Document (from CSA)
- [ ] Access to a Sandbox Workbench Instance for training and evaluation
- [ ] Required knowledge bases curated (from KMO)
- [ ] `hub` CLI installed and configured

> **Note on Workbench Instance and Kubernetes Namespace**: Hub resources do not use "namespace" as a concept. All `hub` commands are scoped to a **workbench instance**. Each workbench instance is mapped to exactly one Kubernetes namespace (n:1 relationship), configured by the Tenant Admin at workbench instance creation time. This mapping is a one-time choice and cannot be changed after deployment. Developers do not need to specify namespace—commands are automatically scoped to the workbench instance.

## Phase 1: Conceive (Define the Need)

> **Your Role Here**: As an Agent Engineer, you typically **receive** the Scenario Definition from a Process Architect. Your job in this phase is to **understand and verify** the inputs, not create them from scratch.

### The Human Analogy

Before hiring someone, organizations identify the need: What role needs to be filled? What team will they join? What responsibilities will they have?

Similarly, before building an agent, you must identify the Scenario that requires automation. **You never start with "I want to build an agent" — you start with "This Scenario needs an agent."**

### 1.1 Review the Scenario

Every agent begins as a response to a Scenario's automation needs. You should have received a `ScenarioNormativeSpec` — review it to understand what your agent must do:

```bash
# View the Scenario Definition you'll be implementing
hub get scenario-normative routine-dispute-triage -o yaml
```

Here's what you're looking for:

```yaml
# The Scenario defines your agent's purpose
scenario:
  name: routine-dispute-triage
  
  # What the agent must help achieve
  goals:
    - "Classify dispute type within 5 minutes"
    - "Request standard documentation automatically"
    - "Approve routine refunds under $100"
  
  # Procedures the agent must follow
  sops:
    - sop-dispute-classification
    - sop-documentation-request
    - sop-refund-approval
  
  # This is what your agent will fulfill
  automationSpec:
    applicationRef:
      name: dispute-triage-agent  # <-- Your agent
      runtime: seer
    role: case-worker
```

### 1.2 Document Provenance

Record where this agent originated—its "birth certificate":

```yaml
metadata:
  name: dispute-triage-agent-v1
  annotations:
    seer.olympus.io/source-scenario: "routine-dispute-triage"
    seer.olympus.io/source-workbench: "acme-disputes"
    seer.olympus.io/genesis-date: "2026-01-11"
    seer.olympus.io/conceived-by: "CSA: disputes-architect@acme.bank"
```

### Checkpoint: Conception Complete

Before proceeding, verify:

- [ ] Scenario Definition exists and is approved
- [ ] Agent's Goals are derived from Scenario Goals
- [ ] Agent's responsibilities are derived from Scenario SOPs
- [ ] Cognitive Design Document is approved by CSA

## Phase 2: Train (Develop in Sandbox)

### The Human Analogy

New employees don't start working on production systems on day one. They go through a **training program** in a **training environment**—learning procedures, practicing with mentors, and making mistakes safely.

Similarly, agents are **first employed in a Sandbox Workbench**—a training environment where they can learn, be evaluated, and evolve without risk to production operations.

### Development Environment

There is no "local development" for Hub applications. All development happens within Hub Workbench Instances:

| Environment | Lifecycle Stage | Use Case |
|-------------|-----------------|----------|
| **Dev Workbench Instance** | `dev` | Rapid iteration, debugging, experimentation |
| **DevOps Workbench** | `dev` or `build` | CI/CD pipelines, automated testing, observability |
| **Sandbox Workbench Instance** | `build` | Pre-production validation, formal testing |

See [DevOps Workbench Pattern](../../../../olympus-hub-docs/09-composite-systems-and-patterns/devops-workbench.md) for setting up your development environment.

### 2.1 Create a Sandbox Workbench Instance

The Sandbox Workbench Instance mirrors production but is safe for experimentation:

```yaml
apiVersion: hub.olympus.io/v1
kind: WorkbenchInstance
metadata:
  name: acme-disputes-sandbox
  labels:
    hub.olympus.io/lifecycle-stage: dev  # Source of truth for lifecycle stage
    hub.olympus.io/workbench: acme-disputes
    hub.olympus.io/tenant: acme
spec:
  workbenchSpec: acme-disputes
  environment: sandbox  # Training environment
  
  # Synthetic or anonymized data
  dataSource:
    type: synthetic
    generator: dispute-synthetic-v1
  
  # Isolated from production systems
  isolation:
    network: sandbox-only
    externalCalls: mocked
```

### 2.2 Create Initial Training Specification

The Training Specification is your agent's "training curriculum".

> **Version Convention**: Use `0.x.x` versions during development (draft). After validation and publication, versions become `1.0.0+` (production-ready). Guardrails become immutable after publication.

```yaml
apiVersion: seer.olympus.io/v1
kind: TrainingSpec
metadata:
  name: dispute-triage-agent-v1
  version: "0.1.0"  # 0.x.x = draft version (pre-publication)
spec:
  # 1. Base capabilities (like hiring someone with relevant education)
  rawAgent:
    name: dispute-analyst-base
    version: "^2.0.0"
  
  # 2. Domain knowledge (like training materials and manuals)
  knowledge:
    sources:
      - type: knowledge-base
        ref: dispute-policies
      - type: knowledge-base
        ref: chargeback-rules
      - type: knowledge-base
        ref: customer-service-guidelines
  
  # 3. Skills and procedures (like SOPs and workflows)
  skills:
    - name: dispute-classification
      procedure: procedures/dispute-classification-sop
    - name: documentation-request
      procedure: procedures/documentation-request-sop
    - name: refund-evaluation
      procedure: procedures/refund-evaluation-sop
  
  # 4. Role definition (like job description)
  prompts:
    system: |
      You are a Dispute Triage Agent operating in the 
      routine-dispute-triage scenario. Your role is to:
      
      1. Classify incoming disputes by type and priority
      2. Request missing documentation from customers
      3. Approve routine refunds under $100
      4. Escalate complex cases to human analysts
      
      You follow the dispute-classification-sop and 
      refund-evaluation-sop procedures precisely.
  
  # 5. Behavioral boundaries (like code of conduct)
  guardrails:
    - ref: financial-limit
      config:
        maxAutonomousAmount: 100
    - ref: pii-protection
    - ref: escalation-required
      config:
        triggers: ["fraud-indicator", "regulatory-dispute"]
  
  # 6. Tools (like systems access)
  tools:
    allowed:
      - protocol: case-lookup
      - protocol: document-request
      - protocol: refund-processor
```

### 2.3 Employ to Sandbox Workbench

Now, **employ** the agent in the sandbox—just as you would assign a trainee to a training program.

> **Note**: You don't create `EmploymentSpec` directly. Instead, you create a `ScenarioDeploymentSpec` targeting the sandbox, and the operators generate the deployment resources.

**Step A: Create ScenarioDeploymentSpec for Sandbox**

```yaml
apiVersion: hub.olympus.io/v1
kind: ScenarioDeploymentSpec
metadata:
  name: routine-dispute-triage-sandbox
  labels:
    hub.olympus.io/workbench-instance: acme-disputes-sandbox
spec:
  automation_ref: routine-dispute-triage
  version: "1.0.0"
  
  workbenchInstance:
    name: acme-disputes-sandbox  # Targets sandbox (lifecycle-stage: dev)
  
  # Sandbox-specific configuration
  task_queues:
    - task_type: triage
      queue_ref: sandbox-disputes
  
  # Training mode settings
  operational:
    humanReviewRate: 1.0  # 100% - All decisions reviewed during training
    observability: full
    decisionLogging: verbose
  
  activation:
    enabled: true
```

**Step B: Deploy Using `hub sync`**

Commit your specs to Git, then sync the Scenario to the sandbox:

```bash
# Commit all specs to Git (GitOps requirement)
git add training-spec.yaml hub-application-spec.yaml scenario-automation-spec.yaml scenario-deployment-sandbox.yaml
git commit -m "feat: initial agent deployment to sandbox"
git push

# Sync the Scenario to deploy (smallest unit of deployment is Scenario)
# Reads all scenario specs from committed Git files
hub sync scenario routine-dispute-triage-sandbox

# Verify deployment status
hub get scenario-deployment routine-dispute-triage-sandbox
```

> **Note**: `hub sync scenario` reads all 3 scenario spec files from Git (ScenarioNormativeSpec, ScenarioAutomationSpec, ScenarioDeploymentSpec) and syncs **all resources** in the scenario atomically.

When synced, the operators automatically generate the deployment resources:

1. **Hub Operators** generate `HubApplicationDeployment` (with sandbox settings)
2. **Seer Operator** generates `EmploymentSpec` (with trainee privileges)

The generated `EmploymentSpec` will include:

```yaml
# GENERATED - Example of what operators create
spec:
  training:
    ref:
      name: dispute-triage-agent-v1
      version: "0.1.0"
  
  workScope:
    workbenchInstance: acme-disputes-sandbox
    lifecycleStage: dev  # Inherited from WorkbenchInstance
    scenario: routine-dispute-triage
  
  authority:
    delegatedBy: training-supervisor@acme.bank
    restrictions:
      - "No production data access"
      - "All decisions logged for review"
```

### 2.4 Iterate Based on Sandbox Experience

Just as human trainees receive feedback and improve, the Training Specification evolves based on sandbox experience:

```
┌─────────────────────────────────────────────────────────────────┐
│                    TRAINING ITERATION LOOP                       │
│                                                                   │
│   ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌────────┐ │
│   │  Employ  │────▶│  Observe │────▶│  Review  │────▶│ Update │ │
│   │  in      │     │  behavior│     │  with    │     │ Training│ │
│   │  Sandbox │     │          │     │  trainer │     │ Spec    │ │
│   └──────────┘     └──────────┘     └──────────┘     └────┬───┘ │
│        ▲                                                   │     │
│        │                                                   │     │
│        └───────────────────────────────────────────────────┘     │
│                          Repeat until                            │
│                          satisfactory                            │
└─────────────────────────────────────────────────────────────────┘
```

**How Iteration Works:**

Since `EmploymentSpec` is generated, here's the actual workflow for iteration:

1. **Observe**: Monitor the Employed Agent's behavior in sandbox
2. **Review**: Identify issues with decisions, escalations, or responses
3. **Update TrainingSpec**: Modify the Training Specification (bump version to 0.x.x)
4. **Update HubApplicationSpec**: Point `seerTrainingRef` to new TrainingSpec version
5. **Reconcile**: Re-sync the Scenario to propagate changes
6. **Repeat**: Continue until behavior is satisfactory

```bash
# After updating TrainingSpec and HubApplicationSpec:
# Edit files, then commit to Git (GitOps requirement)
git add training-spec.yaml hub-application-spec.yaml
git commit -m "feat: update to version 0.2.0"
git push

# Re-sync to propagate changes (reads from Git)
hub sync scenario routine-dispute-triage-sandbox
```

> **Key Insight**: You don't manually recreate the `EmploymentSpec`. When you update the `TrainingSpec` version and the `HubApplicationSpec` references it, the operator chain reconciles automatically on sync.

**How to Monitor Your Agent:**

```bash
# View agent decision logs
hub logs agent dispute-triage-emp-001 --follow

# View agent metrics (response time, escalation rate, etc.)
hub metrics agent dispute-triage-emp-001

# Open the Workbench dashboard (if available)
hub dashboard open acme-disputes-sandbox
```

You can also access agent behavior through the DevOps Workbench — see [DevOps Workbench Pattern](../../../../olympus-hub-docs/09-composite-systems-and-patterns/devops-workbench.md) for observability features.

**Learnings that update the Training Specification:**

| Observation | Training Spec Update |
|-------------|---------------------|
| Agent misclassifies dispute types | Enhance `dispute-classification` skill procedure |
| Agent too aggressive with refunds | Tighten guardrail thresholds |
| Agent missing domain terminology | Add knowledge base entries |
| Agent escalates too frequently | Adjust escalation triggers |
| Agent provides unclear explanations | Refine system prompt |

## Phase 3: Evaluate (Formal Validation)

### The Human Analogy

Before a trainee graduates to full employee status, they undergo **formal evaluation**—skills assessments, competency tests, certification exams. They must demonstrate they can perform the job safely and effectively.

Similarly, before publishing a Training Specification, the agent undergoes **formal validation** against Scenario test cases.

### 3.1 Run Validation Suite

Validation tests the agent against Scenario-derived test cases:

```bash
# Run validation against the deployed sandbox agent
hub validate training-spec dispute-triage-agent-v1 \
  --workbench acme-disputes-sandbox \
  --scenario routine-dispute-triage \
  --test-suites golden-set-disputes,edge-case-disputes,adversarial-prompts,compliance-scenarios

# Or run with a validation config file
hub validate training-spec dispute-triage-agent-v1 -f validation-config.yaml
```

**Example validation config:**

```yaml
# validation-config.yaml
employmentContext:
  workbench: acme-disputes-sandbox
  scenario: routine-dispute-triage
testSuites:
  - golden-set-disputes
  - edge-case-disputes
  - adversarial-prompts
  - compliance-scenarios
evaluationCriteria:
  functionalAccuracy: 0.95
  slaCompliance: 0.99
  guardrailAdherence: 1.0
  escalationAccuracy: 0.98
```

### 3.2 Validation Categories

| Category | What It Tests | Scenario Connection |
|----------|---------------|---------------------|
| **Functional** | Does the agent achieve Scenario Goals? | Goals from Normative Spec |
| **Procedural** | Does the agent follow Scenario SOPs? | SOPs from Normative Spec |
| **Safety** | Do guardrails hold under pressure? | Compliance from Normative Spec |
| **Performance** | Does the agent meet Scenario SLAs? | SLAs from Deployment Spec |
| **Resilience** | Does the agent handle edge cases? | Derived from domain knowledge |

**If validation fails:** Return to Phase 2, update the Training Specification, and re-validate.

## Phase 4: Publish (Graduate for Employment)

### The Human Analogy

When a trainee passes all evaluations, they **graduate** — their training is certified, and they are now **employable**. They can be assigned to any team that needs their skills.

Similarly, when a Training Specification passes validation, it is **published**—making it available for employment in production Workbenches.

### 4.1 Publish Training Specification

```bash
# Publish the validated Training Specification
hub publish training-spec dispute-triage-agent-v1 \
  --version 1.0.0 \
  --approver domain-steward@acme.bank \
  --validation-ref val-dispute-triage-2026-01-11 \
  --notes "Validated in acme-disputes-sandbox. Ready for production employment."

# Verify publication status
hub get training-spec dispute-triage-agent-v1
```

The publication will:
- Bump version from `0.x.x` to `1.0.0`
- Lock guardrails (immutable after publication)
- Make the Training Specification available organization-wide

### 4.2 What Publication Means

| Aspect | Before Publication | After Publication |
|--------|-------------------|-------------------|
| **State** | Draft/Validated | Published |
| **Reusability** | Sandbox only | Any compatible Workbench |
| **Guardrails** | Modifiable | **Immutable** |
| **Version** | 0.x.x | 1.0.0+ (semantic) |
| **Visibility** | Development team | Organization-wide |

### 4.3 Published Training Specification

The published Training Specification is now a **reusable asset**:

```yaml
apiVersion: seer.olympus.io/v1
kind: TrainingSpec
metadata:
  name: dispute-triage-agent
  version: "1.0.0"
  labels:
    seer.olympus.io/status: published
    seer.olympus.io/domain: disputes
    seer.olympus.io/scenario-type: routine-dispute-triage
  annotations:
    seer.olympus.io/validated-in: "acme-disputes-sandbox"
    seer.olympus.io/validation-date: "2026-01-11"
    seer.olympus.io/validation-ref: "val-dispute-triage-2026-01-11"
spec:
  # ... finalized configuration ...
```

## Phase 5: Employ (Assign to Production)

### The Human Analogy

A graduated, skilled employee is **employable**—but they only become **employed** when assigned to a specific team in a specific project or department. At that point, they assume concrete roles, receive actual responsibilities, and are authorized to complete real tasks.

Similarly, a published Trained Agent is **deployable**—but only becomes an **Employed Agent** when bound to a specific Workbench Instance with a concrete environment.

> **Key Terminology**: In Seer, **deploying an agent = employing an agent**. We use "employ" because it accurately describes what happens: the agent is assigned to a work context and granted authority to act.

### 5.1 Create ScenarioDeploymentSpec for Production

To employ an agent to production, you create a `ScenarioDeploymentSpec` targeting the production Workbench Instance. The operators then generate the `HubApplicationDeployment` and `EmploymentSpec`.

> **Reminder**: You don't create `EmploymentSpec` directly. It is generated by the Seer Operator.
> 
> **Note**: Production deployments typically require approval from your Supervisor or ARAO. You may create the spec, but sync is usually performed by authorized personnel.

```yaml
apiVersion: hub.olympus.io/v1
kind: ScenarioDeploymentSpec
metadata:
  name: routine-dispute-triage-prod
  labels:
    hub.olympus.io/workbench-instance: acme-disputes-prod
spec:
  # WHAT: References the validated automation
  automation_ref: routine-dispute-triage
  version: "1.0.0"
  
  # WHERE: Production Workbench Instance
  workbenchInstance:
    name: acme-disputes-prod  # lifecycle-stage: production
  
  # HOW: Production task queues
  task_queues:
    - task_type: triage
      queue_ref: tier-1-disputes
  
  # LIMITS: Production SLAs
  sla:
    target_completion_hours: 24
    escalation_threshold_minutes: 30
  
  # WHEN: Activation settings
  activation:
    enabled: true
    start_date: "2026-02-01"
    schedule: "business-hours"  # When the agent works
```

**What Gets Generated:**

The operators create an `EmploymentSpec` (the agent's "job assignment") with:

```yaml
# GENERATED by Seer Operator - Illustrative example
spec:
  training:
    ref:
      name: dispute-triage-agent-v1
      version: "1.0.0"
  
  workScope:
    workbenchInstance: acme-disputes-prod

    delegatedBy: disputes-supervisor@acme.bank

      perCase: "$0.50"
      monthly: "$5000"
```

### 5.2 What Employment Means

| Human Employment | Agent Employment |
|------------------|------------------|
| Employee assigned to department | Agent bound to Workbench Instance |
| Employee joins specific team | Agent enrolled in Scenario |
| Employee receives access credentials | Agent granted tool bindings |
| Employee gets office/workspace | Agent allocated compute resources |
| Employee reports to manager | Agent has accountable human |
| Employee works defined hours | Agent has temporal scope |
| Employee handles assigned tasks | Agent picks up from task queue |

### 5.3 Deploy (Employ) the Agent

When you apply the `ScenarioDeploymentSpec` and sync, the deployment pipeline activates:

```bash
# Commit the production deployment spec to Git (GitOps requirement)
git add scenario-deployment-prod.yaml
git commit -m "feat: production deployment spec"
git push

# Sync the Scenario to production (requires appropriate permissions)
# Reads all scenario specs from committed Git files
hub sync scenario routine-dispute-triage-prod

# Monitor deployment progress
hub get scenario-deployment routine-dispute-triage-prod --watch
```

```
ScenarioDeploymentSpec (you create)
        │
        ▼ hub sync triggers
HubApplicationDeployment (generated by Hub Operators)
        │
        ▼ Seer Operator processes
EmploymentSpec (generated)
        │
        ▼ Seer Operator creates
Atlantis Resources (Service, Deployment)
```

**Approval Flow** (if configured for production):

```bash
# Request approval for production deployment
hub request-approval scenario-deployment routine-dispute-triage-prod \
  --approver are@acme.bank \
  --production-readiness-ref prc-dispute-triage-2026-01-11 \
  --notes "Approved for production employment"

# Check approval status
hub get approval-requests
```

The agent is now **Employed**—operating in production, handling real work.

```bash
# Verify the employed agent is running
hub get employed-agents

# Check agent health
hub health agent dispute-triage-emp-001
```

### 6.2 Governance of Learnings

> **Critical**: Learnings from Employed Agents do not automatically update Training Specifications. They go through **change management controls** appropriate to the type of change.

| Learning Type | Governance Level | Approver |
|---------------|------------------|----------|
| Prompt refinements | Light | Agent Engineer |
| Knowledge additions | Medium | KMO + Domain Steward |
| Skill procedure updates | Medium | CSA + AE |
| Guardrail changes | **Heavy** | ARAO + Security |
| Tool permission changes | **Heavy** | Security + CSA |

### Lifecycle Stage Labels

The `hub.olympus.io/lifecycle-stage` label is applied to **Workbench Instance** resources only (not all Hub resources):

| Stage | Description | Typical Use |
|-------|-------------|-------------|
| `dev` | Development | Local testing, debugging |
| `build` | Build/CI | Automated testing, validation |
| `staging` | Pre-production | Final validation, UAT |
| `production` | Production | Live operations |

**Sandbox Workbench Instances** typically use `dev` or `build`:

```yaml
# WorkbenchInstance CRD - Source of truth for lifecycle-stage
apiVersion: hub.olympus.io/v1
kind: WorkbenchInstance
metadata:
  name: acme-disputes-sandbox
  labels:
    hub.olympus.io/lifecycle-stage: dev  # ← Source of truth
    hub.olympus.io/workbench: acme-disputes
    hub.olympus.io/tenant: acme
spec:
  workbenchRef: acme-disputes
  environment: sandbox
```

Child resources (`ScenarioDeploymentSpec`, `HubApplicationDeployment`, `EmploymentSpec`) that target this instance inherit the lifecycle-stage.

## Complete Example: All CRDs in Sequence

### Context

| Aspect | Value |
|--------|-------|
| **Scenario** | routine-dispute-triage |
| **Workbench** | acme-disputes |
| **Goals** | Classify disputes, request docs, approve refunds <$100 |
| **Sandbox Instance** | acme-disputes-sandbox (lifecycle-stage: dev) |
| **Production Instance** | acme-disputes-prod (lifecycle-stage: production) |

### Step 1: ScenarioNormativeSpec (Process Architect)

```yaml
apiVersion: hub.olympus.io/v1
kind: ScenarioNormativeSpec
metadata:
  name: routine-dispute-triage
  labels:
    hub.olympus.io/workbench: acme-disputes
spec:
  display_name: "Routine Dispute Triage"
  version: "1.0.0"
  
  roles:
    - name: dispute-triage-agent
      type: agent
      goals:
        - "Classify dispute within 5 minutes"
        - "Request missing documentation"
        - "Approve routine refunds under $100"
  
  sops:
    - ref: sop-dispute-classification
    - ref: sop-refund-approval
```

### Step 2: TrainingSpec (Agent Engineer)

```yaml
apiVersion: seer.olympus.io/v1
kind: TrainingSpec
metadata:
  name: dispute-triage-agent-v1
  labels:
    seer.olympus.io/resource-type: training-spec
    seer.olympus.io/agent-type: case-worker
  annotations:
    seer.olympus.io/source-scenario: "routine-dispute-triage"
spec:
  rawAgent:
    name: dispute-analyst-base
    version: "^2.0.0"
  
  context:
    identity:
      displayName: "Dispute Triage Agent (Training Spec)"
      role: dispute-triage-agent
      domain: disputes
  
  behavioral:
    systemPrompt: |
      You are a Dispute Triage Agent. Your role is to:
      1. Classify incoming disputes
      2. Request missing documentation
      3. Approve routine refunds under $100
  
  guardrails:
    inline:
      - name: refund-cap
        type: decision-boundary
        config:
          maxAutonomousApproval: 100
```

### Step 3: HubApplicationSpec (Developer)

```yaml
apiVersion: hub.olympus.io/v1
kind: HubApplicationSpec
metadata:
  name: dispute-triage-agent
  labels:
    hub.olympus.io/workbench: acme-disputes
    hub.olympus.io/runtime: seer
    seer.olympus.io/resource-type: trained-agent
spec:
  application:
    name: dispute-triage-agent
    display_name: "Dispute Triage Agent (Trained Agent)"
    version: "1.0.0"

  runtime:
    type: seer

  # CANONICAL: Reference to TrainingSpec
  seerTrainingRef:
    name: dispute-triage-agent-v1
    version: "1.0.0"

  dependencies:
    tools:
      - protocol: case-lookup
      - protocol: refund-processor
```

## Anti-Patterns

### ❌ Building Agent Without Scenario Context

```yaml
# WRONG: No Scenario grounding
metadata:
  name: generic-customer-agent
spec:
  prompts:
    system: "You are a helpful customer service agent"
  # What Scenario? What Goals? What SOPs?
  # This agent has no purpose.
```

### ❌ Skipping Sandbox Training

```
Raw Agent ──▶ Training Spec ──▶ Production Employment
                                        │
                                        └── NO SANDBOX VALIDATION
                                            Agent behavior is unproven!
```

### ❌ Direct Production Deployment

```yaml
# WRONG: First ScenarioDeploymentSpec targets production
apiVersion: hub.olympus.io/v1
kind: ScenarioDeploymentSpec
spec:
  workbenchInstance:
    name: production-workbench  # lifecycleStage: production
  # Never deployed to sandbox first!
  # Agent behavior is unvalidated!
```

### ❌ Employing to Unrelated Scenario

```yaml
# Training developed for Scenario A
seerTrainingRef:
  name: fraud-analyst-v1  # Trained for fraud-investigation scenario

# But ScenarioDeploymentSpec is for unrelated Scenario B
automation_ref: customer-inquiry  # Different scenario!
# Agent will fail - wrong training for the job
```

## Troubleshooting

### Common Error Messages

| Error | Likely Cause | Fix |
|-------|--------------|-----|
| `ScenarioNormativeSpec not found` | Missing prerequisite | Verify Scenario exists: `hub get scenario-normative {name}` |
| `TrainingSpec version mismatch` | HubApplicationSpec references wrong version | Update `seerTrainingRef.version` in HubApplicationSpec |
| `Workbench instance not found` | Sandbox doesn't exist or wrong name | Check: `hub get workbench-instances -n {namespace}` |
| `Insufficient permissions for sync` | Not authorized for this lifecycle stage | Request sync from Supervisor for production deployments |
| `Guardrail validation failed` | Config violates constraints | Review guardrail config; some changes require ARAO approval |

### Issue: Validation Fails with "Context Mismatch"

**Symptom:** Validation reports "agent does not understand domain terminology"

**Cause:** Knowledge bases don't include Scenario-specific vocabulary

**Solution:** Work with KMO to add Scenario-relevant knowledge sources to Training Specification

```bash
# Check what knowledge sources are bound
```

### Issue: Agent Escalates Everything

**Symptom:** Agent never makes autonomous decisions in sandbox

**Cause:** Guardrails too restrictive, or system prompt too cautious

**Solution:** Review guardrail thresholds against Scenario autonomy requirements. Adjust during training phase.

```bash
# View current guardrail configuration
hub get training-spec dispute-triage-agent-v1 -o yaml | grep -A20 guardrails
```

### Issue: Agent Works in Sandbox but Fails in Production

**Symptom:** Same Training Spec, different behavior

**Cause:** Production environment has different tool endpoints, data patterns, or timing constraints

**Solution:** Ensure sandbox environment accurately mirrors production. Consider shadow deployment period.

```bash
# Compare sandbox vs production tool bindings
hub diff scenario-deployment routine-dispute-triage-sandbox routine-dispute-triage-prod
```

### Issue: Sync Hangs or Times Out

**Symptom:** `hub sync` doesn't complete

**Cause:** Operator issues or resource conflicts

**Solution:** Check operator logs and deployment status
