# API Operation

**Model:** Definition Model
**Dimension / Track:** Dimension 6: Ecosystem & Extensibility (Platform)
**Owner:** Product Management (API/Platform), Engineering

## Definition

An API Operation is a named, versioned, contractual programmatic operation within an API Module. It is the atomic unit of programmatic capability that external consumers depend on. Each operation is classified by its interaction pattern (Command, Query, Event, Callback, Batch), and carries pattern-appropriate SLOs as strategic performance commitments.

API Operations are the Dim 6 analog of User Journeys in Dim 4 — purposeful, named interactions that consumers depend on. Specific URL paths, HTTP methods, Kafka topic names, and payload schemas are PSD and Build Track territory; the Definition Model captures the operation's identity, pattern, and commitments.

## Purpose

Captures what the product's programmatic surface commits to — the named operations consumers can depend on, their interaction patterns, and performance guarantees. Enables impact analysis (if this operation changes, who breaks?), SLO-driven design, and contractual clarity with Developer and Programmatic User Personas.

## Fields

| Field | Type | Description |
|---|---|---|
| Name | Text | Operation name (e.g., "Create Payment," "payment.settled," "Upload Payment File") |
| Version | Text | Operation version (inherits from or overrides API Module version) |
| Pattern | Enum | Command / Query / Event / Callback / Batch |
| Direction | Enum | Consumer → Product / Product → Consumer / Bidirectional |
| Description | Text | What this operation does |
| Idempotent | Boolean | Whether repeated invocations produce the same result |
| SLOs | Structured | Pattern-appropriate performance commitments (see below) |

### SLO Fields by Pattern

| Pattern | SLO Fields |
|---|---|
| **Command** | Availability (%), Latency (p50, p95, p99), Throughput (requests/sec), Error Budget |
| **Query** | Availability (%), Latency (p50, p95, p99), Throughput (requests/sec), Error Budget |
| **Event** | Delivery Guarantee (at-least-once / exactly-once), Delivery Latency (trigger-to-delivery), Ordering Guarantee (ordered / unordered), Retry Semantics |
| **Callback** | Delivery Guarantee, Delivery Latency, Retry Semantics |
| **Batch** | Processing Window (max time from submission to completion), Throughput (records/time), Completeness (partial success handling), Scheduling (periodic / on-demand) |

## Statuses

| Status | Description |
|---|---|
| Draft | Operation defined but not yet published |
| Published | Operation available for external consumption |
| Deprecated | Operation marked for retirement; sunset period active |
| Retired | Operation no longer available |

## Example

**Command — "Create Payment"**
Pattern: Command. Direction: Consumer → Product. Idempotent: Yes (via idempotency key). SLOs: 99.95% availability, p99 < 500ms, 10K calls/min throughput. Maps to: Payment Initiation capability (Payment Processing Module). Description: Submits a cross-border payment instruction for processing.

**Event — "payment.settled"**
Pattern: Event. Direction: Product → Consumer. SLOs: at-least-once delivery, < 30s delivery latency, ordered per payment ID. Maps to: Payment Settlement capability. Description: Emitted when a cross-border payment reaches final settlement.

**Batch — "Upload Payment File"**
Pattern: Batch. Direction: Consumer → Product. SLOs: processing within 4 hours, 100K records/batch max, partial success with rejection report. Maps to: Batch Payment Processing capability. Description: Accepts a batch file of payment instructions via SFTP for bulk processing.

**Batch — "Daily Settlement Report"**
Pattern: Batch. Direction: Product → Consumer. SLOs: generated by 06:00 UTC daily, complete for all settlements in previous calendar day. Maps to: Settlement Reporting capability. Description: Generates a daily SFTP file of all settled payments for reconciliation.
