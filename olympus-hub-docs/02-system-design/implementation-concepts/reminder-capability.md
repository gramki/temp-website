# Reminder Capability

> **Category:** Signal Architecture

---

## Overview

The **Reminder Capability** is Signal Exchange's built-in time-based stimulus feature. Hub Applications can schedule reminders that trigger at specified times, enabling workflows to continue when external inputs don't arrive. This eliminates the need for applications to maintain their own schedulers for timeout handling.

---

## Ontology Context

### Relationship to Ontology

The ontology describes **Signals** as stimuli that can be external or internal. Reminder is an internal, time-based signal generated by the platform on behalf of applications.

| Ontology Concept | Implementation | Relationship |
|------------------|----------------|--------------|
| Internal Signal | Reminder Notification | Platform-generated time-based signal |
| (not covered) | Reminder scheduling | SX-managed timer capability |

### Gap This Fills

The ontology describes signals abstractly. Reminder Capability addresses:
1. **Timeout handling**: What happens when expected signals don't arrive?
2. **Workflow continuation**: How do parked workflows resume?
3. **Scheduler management**: Who manages time-based triggers?

---

## Definition

**Reminder Capability** is an SX-provided feature that:
- Allows applications to schedule future stimuli
- Delivers REMINDER_NOTIFICATION at scheduled time
- Supports one-time and recurring reminders
- Includes context for application to understand the reminder
- Automatically cancels reminders when requests complete

### Characteristics

| Characteristic | Description |
|----------------|-------------|
| **Scope** | Request-scoped; reminders tied to requests |
| **Lifecycle** | Created by application; delivered by SX; cancelled on completion |
| **Ownership** | SX manages; applications consume |
| **Multiplicity** | Multiple reminders per request |

---

## Rationale

### Why This Design?

Built-in reminders simplify application design:
1. **No external schedulers**: Applications don't need to manage timers
2. **Request-aware**: Reminders cancelled when request completes
3. **Reliable delivery**: SX guarantees delivery (at-least-once)
4. **Consistent interface**: Same envelope format as other messages

### Alternatives Considered

| Alternative | Why Not Chosen |
|-------------|----------------|
| **External scheduler per app** | Complexity; duplication; reliability |
| **Polling for timeouts** | Inefficient; delays; resource waste |
| **No timeout handling** | Workflows get stuck |

## Structure

### Reminder Request

```json
{
  "update_type": "REMIND",
  "payload": {
    "reminder": {
      "reminder_schedule_id": "rem-doc-upload-48h",
      "kind": "document_upload_timeout",
      "schedule": {
        "type": "once",              // once | recurring
        "when": "2026-01-08T10:00:00Z",
        "cron_expression": null      // For recurring
      },
      "reminder_payload": {
        "content_type": "application/json",
        "semantic_type": "com.acme.DocumentReminder",
        "data": {
          "message": "Customer has not uploaded documents",
          "task_id": "task-abc",
          "agent_id": "agent-xyz"
        }
      }
    }
  }
}
```

### Reminder Notification

```json
{
  "update_type": "REMINDER_NOTIFICATION",
  "payload": {
    "reminder_schedule_id": "rem-doc-upload-48h",
    "kind": "document_upload_timeout",
    "sequence_in_schedule": 1,
    "scheduled_at": "2026-01-08T10:00:00Z",
    "triggered_at": "2026-01-08T10:00:01Z",
    "original_request_payload": {
      // Full original request payload
    },
    "reminder_payload": {
      // Payload from reminder request
    }
  }
}
```

### Reminder Kind

The `kind` attribute provides semantic context:

| Kind | Purpose |
|------|---------|
| `document_upload_timeout` | Customer hasn't uploaded required documents |
| `response_timeout` | External system hasn't responded |
| `sla_warning` | Approaching SLA deadline |
| `follow_up` | Scheduled follow-up action |
| `escalation_trigger` | Time for escalation |

## Behavior

### How It Works

**Scheduling a Reminder:**
```
1. Application parses request, determines wait needed
2. Application sends REMIND envelope to SX:
   ├── reminder_schedule_id (unique for this reminder)
   ├── kind (semantic type)
   ├── schedule (when to trigger)
   └── reminder_payload (context for handling)
3. SX stores reminder against request
4. SX acknowledges reminder scheduled
```

**Receiving Reminder:**
```
1. Scheduled time arrives
2. SX creates REMINDER_NOTIFICATION envelope:
   ├── Original request payload
   ├── Reminder payload from schedule
   ├── Trigger metadata
   └── Sequence number in schedule
3. SX delivers to Application
4. Application processes and takes action
```

**Automatic Cancellation:**
```
Request reaches terminal state (COMPLETED, CANCELLED):
├── All associated reminders cancelled
├── No REMINDER_NOTIFICATION sent
└── If reminder triggers during completion, suppressed
```

### Use Cases

| Scenario | Reminder Use |
|----------|--------------|
| **Awaiting documents** | Remind after 48h to take alternate path |
| **External API call** | Reminder if no callback within 1h |
| **SLA tracking** | Warn at 80% of SLA time |
| **Recurring check** | Daily status check while in progress |

### Interactions

| Interacts With | Direction | Description |
|----------------|-----------|-------------|
| Signal Exchange | ↔ managed by | SX stores and triggers reminders |
| Hub Application | ↔ schedules/receives | App schedules; SX delivers |
| Request | → bound to | Reminders cancelled with request |

## Constraints and Invariants

| Constraint | Description |
|------------|-------------|
| **Request-scoped** | Reminders tied to specific request |
| **Unique schedule ID** | reminder_schedule_id unique within request |
| **Future time** | Reminder time must be in future |
| **Cancelled on completion** | Terminal request cancels all reminders |
| **At-least-once** | SX guarantees delivery (may duplicate) |

---

## Pros and Cons

### Benefits

| Benefit | Description |
|---------|-------------|
| ✅ **Simple application logic** | No scheduler management |
| ✅ **Automatic cleanup** | Cancelled on request completion |
| ✅ **Reliable delivery** | SX handles retries |
| ✅ **Semantic context** | `kind` enables smart handling |

### Trade-offs

| Trade-off | Mitigation |
|-----------|------------|
| ⚠️ **Time precision** | "At or shortly after" semantics |
| ⚠️ **Duplicate delivery possible** | Idempotent reminder handling |

## Examples

### Example 1: Document Upload Timeout

```python
# Application waiting for customer documents
async def handle_request(self, envelope):
    # Create task for document upload
    task = await self.create_task(
        task_type="upload_documents",
        target="customer"
    )
    
    # Schedule reminder in case documents don't arrive
    await self.send_update({
        "update_type": "REMIND",
        "payload": {
            "reminder": {
                "reminder_schedule_id": f"doc-reminder-{task.id}",
                "kind": "document_upload_timeout",
                "schedule": {
                    "type": "once",
                    "when": (now + timedelta(hours=48)).isoformat()
                },
                "reminder_payload": {
                    "data": {
                        "task_id": task.id,
                        "fallback_action": "proceed_without_documents"
                    }
                }
            }
        }
    })

async def handle_reminder(self, envelope):
    if envelope.payload.kind == "document_upload_timeout":
        # Documents not received, take alternate path
        await self.proceed_without_documents(
            envelope.payload.reminder_payload.data
        )
```

## Implementation Notes

### For Developers

- Use meaningful `kind` values for filtering
- Include enough context in reminder_payload
- Handle duplicate REMINDER_NOTIFICATION idempotently
- Cancel reminders explicitly if alternate path taken

### For Operators

- Monitor reminder queue depth
- Check for orphaned reminders (request completed but reminder exists)
- Review reminder trigger latency
