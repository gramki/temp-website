# Hub Application Deployment

> **Category:** Configuration Model

---

## Overview

A **Hub Application Deployment** is a CRD that represents a running instance of a Hub Application within a specific Workbench Instance. It is **generated by Hub Operators** when a `ScenarioDeploymentSpec` is processed, and serves as the parent resource for runtime-specific child resources (e.g., Seer `EmploymentSpec`, Atlantis `Service`/`Deployment`).

---

## Ontology Context

### Relationship to Ontology

The ontology defines **Automation** as the blueprint and **Operation** as the running instance. Hub Application Deployment is the bridge:

| Ontology Concept | Implementation | Relationship |
|------------------|----------------|--------------|
| Automation | HubApplicationSpec | Blueprint/template |
| Operation Instance | HubApplicationDeployment | Running deployment |
| Agent (Employed) | EmploymentSpec (Seer child) | Runtime authority |

### Gap This Fills

`HubApplicationSpec` defines *what* an application is. `HubApplicationDeployment` specifies:
1. **Instance Identity**: Which deployment of the application
2. **Runtime Context**: Which Workbench Instance and lifecycle stage
3. **Runtime Extensions**: Runtime-specific configuration (Seer, Rhea, Atlantis)
4. **Parent-Child Relationship**: Anchor for runtime-created resources

---

## Definition

**Hub Application Deployment** is a CRD that:
- Is **generated by Hub Operators** from `ScenarioDeploymentSpec`
- Represents a **specific deployment** of a `HubApplicationSpec`
- Contains **runtime-specific extensions** for the target runtime
- Acts as **parent resource** for runtime-created child resources
- Tracks **deployment state** and child resource status

### Characteristics

| Characteristic | Description |
|----------------|-------------|
| **Scope** | Workbench Instance-scoped |
| **Lifecycle** | Created by Hub Operators; managed by runtime operators |
| **Ownership** | Hub Operators create; Runtime Operators add children |
| **Kubernetes Namespace** | Each workbench instance maps to one Kubernetes namespace (n:1), configured by Tenant Admin at deployment time (one-time choice) |
| **Multiplicity** | One per ScenarioDeploymentSpec per application |

## Naming Conventions

### Resource Naming

| Pattern | Example | Description |
|---------|---------|-------------|
| `{app-name}-{instance-short}-{seq}` | `dispute-triage-agent-prod-001` | Standard deployment |
| `{app-name}-{tenant}-{stage}-{seq}` | `dispute-triage-acme-dev-001` | Multi-tenant clarity |

### Display Name Conventions

| Context | Display Name Pattern |
|---------|---------------------|
| **Generic** | `"{App Name} (Deployed) - {Instance}"` |
| **Seer Trained Agent** | `"{Agent Name} (Employed) - {Instance}"` |

### Labels (Required)

```yaml
labels:
  hub.olympus.io/tenant: acme
  hub.olympus.io/workbench: acme-disputes
  hub.olympus.io/workbench-instance: acme-disputes-prod
  hub.olympus.io/scenario: routine-dispute-triage
```

Note: `hub.olympus.io/lifecycle-stage` is inherited from the parent `WorkbenchInstance`, not set directly on `HubApplicationDeployment`.

### Labels (Seer-Specific)

```yaml
labels:
  seer.olympus.io/resource-type: employed-agent
  seer.olympus.io/agent-type: case-worker
```

## CRD Schema

```yaml
apiVersion: hub.olympus.io/v1
kind: HubApplicationDeployment
metadata:
  name: dispute-triage-agent-acme-prod-001
  labels:
    # Required labels
    hub.olympus.io/tenant: acme
    hub.olympus.io/workbench: acme-disputes
    hub.olympus.io/workbench-instance: acme-disputes-prod
    hub.olympus.io/scenario: routine-dispute-triage
    # Inherited from WorkbenchInstance (not set directly)
    # hub.olympus.io/lifecycle-stage: production
    # Seer-specific (for Trained Agents)
    seer.olympus.io/resource-type: employed-agent
    seer.olympus.io/agent-type: case-worker
  annotations:
    hub.olympus.io/generated-from: "ScenarioDeploymentSpec/routine-dispute-triage-prod"
    hub.olympus.io/application-spec: "HubApplicationSpec/dispute-triage-agent"
    # Seer-specific
    seer.olympus.io/training-ref: "TrainingSpec/dispute-triage-agent-v1"

spec:
  # ============================================================
  # IDENTITY
  # ============================================================
  deployment:
    name: dispute-triage-agent-acme-prod-001
    display_name: "Dispute Triage Agent (Employed) - ACME Production"
    version: "1.0.0"
    description: "Production deployment of dispute triage agent"

  # ============================================================
  # SOURCE REFERENCES
  # ============================================================
  applicationSpecRef:
    name: dispute-triage-agent
    version: "1.0.0"

  scenarioDeploymentRef:
    name: routine-dispute-triage-prod

  # ============================================================
  # WORKBENCH INSTANCE CONTEXT
  # ============================================================
  workbenchInstance:
    name: acme-disputes-prod
    lifecycleStage: production  # dev | build | staging | production

  # ============================================================
  # RUNTIME CONFIGURATION
  # ============================================================
  runtime:
    type: seer  # seer | rhea | atlantis | chronoshift | perseus

    # Seer Runtime Extension (for AI Agents)
    seerExtension:
      # Reference to TrainingSpec (from HubApplicationSpec.seerTrainingRef)
      trainingSpecRef:
        name: dispute-triage-agent-v1
        version: "1.0.0"

      # Authority delegation (from ScenarioDeploymentSpec)
      delegation:
        accountableHuman: disputes-supervisor@acme.bank
        role:
          name: fraud-analyst

      # Capacity allocation
      capacity:
        compute:
          replicas: 2
          cpu: "2"
          memory: "4Gi"
        tokens:
          daily: 1000000
          perRequest: 50000

    # Rhea Runtime Extension (for Workflows)
    rheaExtension:
      workflowVersion: "1.0.0"
      taskQueues:
        - name: dispute-workflow-queue

    # Atlantis Runtime Extension (for Containers)
    atlantisExtension:
      image: "registry.hub.olympus.io/acme/dispute-handler:1.0.0"
      replicas: 3

  # ============================================================
  # RESOLVED BINDINGS
  # ============================================================
  resolvedBindings:
    # Tool bindings (operative - resolved from normative refs)
    tools:
      - protocol: case-lookup
        machine: acme-core-banking
        instance: acme-get-transactions
        endpoint: https://core.acme.bank/api/v1/transactions

      - protocol: refund-processor
        machine: acme-payments
        instance: acme-refund-api
        endpoint: https://payments.acme.bank/api/v1/refunds

    # Resource bindings
    resources:
      - name: dispute-policies
        type: knowledge-bank
        instance: acme-dispute-kb-prod

    # Memory bindings (for Seer)
    memory:
      - name: case-dialog
        store: disputes-conversation-store
        storeId: conv-store-001

status:
  # Deployment state
  state: Active  # Pending | Deploying | Active | Suspended | Terminated
  
  # Timestamps
  createdAt: "2026-01-11T10:00:00Z"
  deployedAt: "2026-01-11T10:02:00Z"
  
  # Runtime-specific status (filled by runtime operator)
  runtimeStatus:
    # For Seer: EmploymentSpec reference
    employmentSpecRef:
      name: dispute-triage-emp-001
    # Child resources created by runtime operator
    childResources:
      - apiVersion: seer.olympus.io/v1
        kind: EmploymentSpec
        name: dispute-triage-emp-001
      - apiVersion: v1
        kind: Service
        name: dispute-triage-emp-001-svc
      - apiVersion: apps/v1
        kind: Deployment
        name: dispute-triage-emp-001-deployment

  # Health
  health:
    ready: true
    availableReplicas: 2
    desiredReplicas: 2

  # Conditions
  conditions:
    - type: Ready
      status: "True"
      lastTransitionTime: "2026-01-11T10:02:00Z"
    - type: RuntimeReady
      status: "True"
      lastTransitionTime: "2026-01-11T10:02:00Z"
      message: "Seer EmploymentSpec active"
```

### State Transitions

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Pending   │────►│  Deploying  │────►│   Active    │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                         ┌─────────────────────┼─────────────────────┐
                         ▼                     │                     ▼
                  ┌─────────────┐              │              ┌─────────────┐
                  │  Suspended  │◄─────────────┘              │ Terminated  │
                  └──────┬──────┘                             └─────────────┘
                         │
                         └──────────► (can return to Active)
```

| State | Description | Runtime Active? |
|-------|-------------|-----------------|
| **Pending** | Created, awaiting runtime operator | ❌ |
| **Deploying** | Runtime operator processing | ❌ |
| **Active** | Deployed and serving | ✅ |
| **Suspended** | Temporarily paused | ❌ |
| **Terminated** | Permanently stopped | ❌ |

## Runtime-Specific Behavior

### Seer Runtime

When `runtime.type: seer`:

1. **Seer Operator** watches for new `HubApplicationDeployment`
2. Creates `EmploymentSpec` with:
   - `ownerReference` pointing to `HubApplicationDeployment`
   - Training reference from `seerExtension.trainingSpecRef`
   - Resolved bindings from `resolvedBindings`
3. Creates Atlantis resources (Service, Deployment)
4. Updates `status.runtimeStatus.employmentSpecRef`

### Rhea Runtime

When `runtime.type: rhea`:

1. **Rhea Operator** watches for new `HubApplicationDeployment`
2. Creates workflow engine resources
3. Configures task queues
4. Updates status with workflow instance reference

### Atlantis Runtime

When `runtime.type: atlantis`:

1. **Atlantis Operator** watches for new `HubApplicationDeployment`
2. Creates Kubernetes Service and Deployment
3. Configures ingress and scaling
4. Updates status with pod references

## Lifecycle Stage Labels

The `hub.olympus.io/lifecycle-stage` label is defined on **WorkbenchInstance** resources. `HubApplicationDeployment` resources **inherit** this label from the WorkbenchInstance they are deployed to:

| Stage | Description | Typical Use |
|-------|-------------|-------------|
| `dev` | Development environment | Local testing, debugging |
| `build` | Build/CI environment | Automated testing, validation |
| `staging` | Pre-production | Final validation, UAT |
| `production` | Production | Live operations |

**Sandbox Workbench Instances** typically use `dev` or `build` stages. Deployments to those instances inherit the stage:

```yaml
# WorkbenchInstance defines the lifecycle stage
apiVersion: hub.olympus.io/v1
kind: WorkbenchInstance
metadata:
  name: acme-disputes-sandbox
  labels:
    hub.olympus.io/lifecycle-stage: dev  # Source of truth

---
# HubApplicationDeployment inherits from WorkbenchInstance
metadata:
  labels:
    hub.olympus.io/lifecycle-stage: dev  # Inherited
    hub.olympus.io/workbench-instance: acme-disputes-sandbox
```
